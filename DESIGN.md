# CostIQ Panel Design

Дата: 2026-05-22
Статус: обязательный дизайн-контекст для UI-изменений панели

## Назначение

Этот файл фиксирует визуальный стандарт CostIQ Panel для AI-агентов и разработчиков. Его нужно читать перед любыми изменениями интерфейса панели, особенно для новых экранов, карточек, калькуляторов, ИИ-рабочего стола, форм заявок и админских инструментов.

Цель: панель должна оставаться рабочим кабинетом САУ, а не набором случайных экранов в стиле типового AI/Tailwind-шаблона.

## Продуктовый Образ

CostIQ Panel - это деловой инструмент для ежедневной работы с ОТ, КП, ПС, ВОР, сметными справочниками, проектами, заявками, калькуляторами и документами.

Интерфейс должен быть:

- плотным, но читаемым;
- спокойным и профессиональным;
- ориентированным на повторяемую работу, а не на презентационный эффект;
- удобным в Telegram WebView на телефоне и в браузере на десктопе;
- быстрым для сканирования статусов, сумм, отклонений, файлов и действий.

Не делать лендинговую страницу, маркетинговый hero или декоративный промо-экран там, где нужен рабочий инструмент.

## Базовая Палитра

Использовать текущие CSS-переменные как источник истины:

- фон: #111214;
- основная поверхность: #1b1d21;
- вторичная поверхность: #23262c;
- третичная поверхность: #2a2e35;
- линии и рамки: #343841;
- основной текст: #f2f4f7;
- вторичный текст: #aab1bd;
- акцент успеха: #59d18c;
- информационный акцент: #6ea8ff;
- медный акцент ФСК: #f18a45;
- ошибка / риск: #ff7a7a;
- доп. акценты: #58d2c4 и #b69cff.

Правила:

- темный фон остается базовым;
- медный акцент использовать дозированно для важных действий, обновлений и FSK-идентичности;
- не превращать экран в одноцветную зеленую, синюю или медную тему;
- не добавлять декоративные градиентные шары, блики, bokeh и нерабочие фоны;
- цвет не должен быть единственным носителем смысла: статус всегда поддерживать текстом.

## Типографика

- Шрифт: Arial, Helvetica, sans-serif.
- Letter spacing: 0.
- Hero-scale заголовки использовать только для главного приветственного экрана. В рабочих карточках и панелях заголовки должны быть компактными.
- Текст должен помещаться в контейнеры на мобильном и десктопе.
- Длинные названия файлов, trace_id, R2 keys и локальные пути не показывать пользователю. Использовать короткие человеческие имена результата, например Итог_ОТ_06669.zip.

## Сетка И Пространство

- Основная ширина: до 1180 px для админского режима и до 960 px для публичного режима.
- Внешний отступ shell: 16 px по бокам, с учетом safe-area.
- Базовый gap: 8-14 px.
- Радиус карточек и контролов: 8 px или меньше, кроме pill-статусов и навигации.
- Не вкладывать карточки в карточки без функциональной причины.
- Разделы должны быть рабочими блоками, а не декоративными floating cards.
- Для fixed-format элементов задавать стабильные размеры, grid tracks, min/max и aspect-ratio, чтобы hover, loading и длинные подписи не двигали сетку.

## Навигация

Верхнее меню остается основным способом движения:

- Главная;
- Навыки;
- Калькуляторы;
- Инструменты;
- Поддержка;
- Мои заявки.

Правила:

- навигация липкая, горизонтальная, прокручиваемая на мобильном;
- не дублировать основные входы на главной, если они уже есть в верхнем меню;
- активный раздел выделять спокойной рамкой и фоном, без агрессивных эффектов;
- новые инструменты добавлять в существующие разделы, а не плодить верхние вкладки без необходимости.

## Компоненты

### Карточки

Использовать карточки для повторяемых сущностей:

- заявки;
- навыки;
- инструменты;
- результаты;
- новости;
- калькуляторные сценарии;
- элементы очереди;
- карточки Agent Factory.

Карточка должна иметь:

- короткий заголовок;
- одну строку сути;
- статус или метрику;
- понятное основное действие;
- минимум вторичного шума.

### Кнопки

- Основное действие: спокойная акцентная кнопка.
- Вторичное действие: ghost/button с рамкой.
- Опасное действие: красный акцент только для реального риска.
- Иконки или короткие буквенные маркеры допустимы, но не должны заменять понятное действие там, где нужна точность.
- Не делать длинные текстовые кнопки, которые ломаются в Telegram WebView.

### Формы

- Label обязателен.
- Placeholder не заменяет label.
- Ошибку показывать рядом с формой или полем понятным текстом.
- Для загрузки файла показывать имя файла, размер и понятный лимит.
- Для черновиков использовать localStorage, если сценарий может быть прерван Telegram WebView.

### Статусы

Статусы должны быть человеко-понятными:

- создана;
- в работе;
- готово к проверке;
- вопрос отправлен;
- нужна доработка;
- принято;
- закрыто;
- ошибка.

Технические статусы допускаются в админке, но рядом должен быть смысловой перевод.

## Таблицы И Списки

- На мобильном избегать широких таблиц; использовать карточки или компактные списки.
- Для админки допустимы плотные таблицы, если они не ломают viewport.
- Суммы и проценты выравнивать так, чтобы их можно было сравнивать глазами.
- Для длинных списков добавлять фильтры, поиск, статусы и пустые состояния.

## Мои Заявки И Результаты

Карточка результата должна показывать:

- статус;
- версию результата;
- текущий этап;
- короткое имя файла;
- краткий текст итога;
- действия: задать вопрос, нужна доработка, принять результат.

Запрещено:

- показывать trace_id как главное имя результата;
- показывать R2/local storage key;
- показывать длинные машинные пути в тексте;
- скрывать причину ошибки за общим словом "ошибка".

## Калькуляторы

Раздел Калькуляторы должен быть рабочим экраном быстрых расчетов, а не ссылкой на очередь.

MVP быстрых расчетов:

- НДС: начислить, выделить, ставка 20%, 22%, пользовательская;
- проценты: процент от числа, X это сколько процентов от Y, прибавить процент, вычесть процент;
- отклонение КП / ПС / ГТ / медиана: рубли и проценты;
- цена за единицу: сумма / объем = руб./м2, руб./м3, руб./м.п., руб./шт.;
- обратный расчет объема: сумма / ставка;
- округление и копирование результата.

ПТО-слой:

- бетон;
- арматура;
- земляные работы;
- отделка;
- фасады;
- воздуховоды;
- кабель, трубы, лотки;
- перевод единиц.

Правило: простые расчеты делать на frontend без bridge и очереди. Bridge подключать только для сохранения в объект, формирования Excel, создания заявки или вопроса к ИИ по результату.

## ИИ-Рабочий Стол

ИИ-рабочий стол должен быть отдельным рабочим местом, а не обычным чатом.

Рекомендуемая структура:

- слева: контекст объекта, договора, подрядчика, заявки или файла;
- центр: диалог, анализ, таблица или результат;
- справа: быстрые калькуляторы, справочники, история, действия.

Режимы:

- проверить;
- объяснить;
- сравнить;
- найти риски;
- сделать письмо;
- сделать замечания;
- сделать сводку;
- создать заявку из ответа.

Дизайн должен поддерживать работу с файлами, таблицами, объектами и историей, а не только свободный текстовый промпт.

## Инструменты

Раздел Инструменты должен открывать реальные рабочие экраны:

- Сметный справочник;
- НЦС / УПСС;
- ТЭП проекта;
- Удельные показатели;
- Agent Factory;
- будущие интерактивные инструменты САУ/ПТО.

Карточка инструмента должна показывать вход, выход, текущий статус, основной сценарий и быстрые действия.

## Админка

Админские экраны могут быть плотнее публичных, но должны оставаться читаемыми.

Обязательные свойства:

- видимые состояния очереди;
- ошибки и retry без скрытия деталей;
- operator events и checkpoints;
- защищенные действия с токеном;
- отсутствие секретов в публичном DOM и ответах API.

## Telegram WebView

Проверять интерфейс как мобильный рабочий инструмент.

Правила:

- не полагаться только на browser submit, если Telegram WebView уже ломал сценарий;
- важные кнопки должны иметь явные click-handlers;
- не использовать модальные окна, которые перекрывают основной процесс и не закрываются на мобильном;
- учитывать safe-area и маленькую высоту экрана;
- не делать горизонтальные элементы без overflow-x;
- кэш статики для актуальных JS/CSS должен контролироваться версией и no-store там, где это критично.

## Пустые, Ошибочные И Загрузочные Состояния

Каждый новый экран должен иметь:

- загрузку;
- пустое состояние;
- ошибку с понятной причиной;
- повтор действия, если это применимо;
- fallback для браузера вне Telegram.

Нельзя оставлять пустой блок без объяснения.

## Запреты

Не использовать:

- лендинговые hero-блоки для рабочих экранов;
- декоративные орбы, bokeh, случайные градиентные фоны;
- длинные машинные имена файлов;
- локальные пути, storage keys и trace_id как пользовательские названия;
- огромные карточки без плотной рабочей пользы;
- вложенные карточки ради визуального эффекта;
- широкие псевдотаблицы в публичном мобильном интерфейсе;
- анимации, которые мешают проверке суммы, статуса или действия;
- текст "как пользоваться" внутри рабочего интерфейса вместо понятных контролов и состояний.

## Acceptance Для UI-Изменений

Перед завершением UI-задачи проверить:

- экран соответствует темной рабочей стилистике CostIQ;
- на мобильной ширине ничего не перекрывается;
- длинные названия и суммы не ломают карточки;
- основные действия видны без прокрутки там, где это критично;
- статусы понятны человеку;
- результат не показывает внутренние пути и trace_id как название;
- есть loading, empty и error states;
- frontend проходит node --check app.js, если менялся JS;
- CSS не создает одну доминирующую цветовую тему и не добавляет декоративный шум.
